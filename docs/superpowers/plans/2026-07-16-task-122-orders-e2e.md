# TASK-122 — Consulta, detalhe e cancelamento recusado E2E Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Provar no Chromium que um cliente autenticado lista e filtra pedidos conhecidos, abre o detalhe, tenta cancelar com o contrato exato, recebe `422` anunciado e continua vendo o status confirmado pelo servidor após refetch e reload.

**Architecture:** Estender o backend determinístico em memória de `frontend/e2e/support/authApi.ts` com fixtures de pedidos, quatro ledgers estritos (`ordersList`, `orderDetail`, `orderProduct` e `orderCancel`) e estado mutável que registra a tentativa recusada sem alterar o pedido confirmado. A nova spec `frontend/e2e/orders.spec.ts` inicia pelo login, percorre lista, filtro, detalhe e diálogo exclusivamente por seletores semânticos, e fecha a jornada com reload e contagens brutas exatas.

**Tech Stack:** React 19, TypeScript 5.7, TanStack Query 5, Playwright 1.61, Vite 6.

Declare `pendingOrderProduct` no estado isolado do helper, arme-o em cada GET do
detalhe e limpe-o no `reset`. O marcador é consumível e só classifica
`orderProduct` quando o frame corrente ainda está em `/pedidos/:id`. Se o
produto for consultado depois que a jornada navegou para `/produtos/:id`, a
consulta consome o marcador como `product`. Isso cobre o refetch do detalhe com
produto em cache sem deixar contexto cumulativo para requests comuns futuros.

- [ ] **Step 7: Implementar a lista com duas queries exatas**

Antes do handler `POST /api/v1/pedido`, adicione:

```ts
    if (url.pathname === '/api/v1/pedido' && request.method() === 'GET') {
      requireAuthorization(route)
      increment('ordersList')
      if (request.postData() !== null) {
        throw new Error(
          `Expected empty orders list body, received ${request.postData()}`,
        )
      }

      const query = url.searchParams
      const expectedCpf = registeredCustomer?.cpf
      if (!expectedCpf || query.get('cpf') !== expectedCpf) {
        throw new Error(`Unexpected orders CPF: ${query.get('cpf')}`)
      }
      if (query.get('page') !== '1' || query.get('size') !== '20') {
        throw new Error(`Unexpected orders pagination: ${url.search}`)
      }

      const start = query.get('dataInicio')
      const end = query.get('dataFim')
      const isUnfiltered =
        start === null
        && end === null
        && [...query.keys()].sort().join(',') === 'cpf,page,size'
      const isFiltered =
        start === '2026-07-01T03:00:00.000Z'
        && end === '2026-07-16T02:59:59.999Z'
        && [...query.keys()].sort().join(',')
          === 'cpf,dataFim,dataInicio,page,size'

      if (!isUnfiltered && !isFiltered) {
        throw new Error(`Unexpected orders query: ${url.search}`)
      }

      const orders = isFiltered
        ? [targetOrder()]
        : [targetOrder(), secondaryOrder()]
      await json(route, {
        status: true,
        pagination: {
          pages: 1,
          size: 20,
          totalItems: orders.length,
          data: orders,
        },
      })
      return
    }
```

O primeiro GET aceita somente `cpf,page,size`; o segundo aceita somente as cinco chaves e os limites ISO exatos.
